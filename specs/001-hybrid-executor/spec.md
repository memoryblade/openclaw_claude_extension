# 功能规格说明：混合执行器路由

**功能分支**: `001-hybrid-executor`
**创建日期**: 2026-03-26
**状态**: 草稿
**输入**: 用户描述："为 OpenClaw 增加混合执行能力：保留 OpenClaw 作为主控层，日常多渠道对话继续走低成本路径；当用户显式要求使用 Claude Code 或 Codex CLI 时，调用独立的服务，并在同一 session 中持续保持该高级执行器会话，直到用户显式回退。"

## 澄清

### 会话 2026-03-26

- Q: 升级到高级执行器时传递哪些上下文？ → A: 传递当前会话的完整消息历史。
- Q: 从高级执行器回退时是否带回对话内容？ → A: 是，高级执行器的对话内容完整带回默认执行器。
- Q: 高级执行器间直接切换时是否传递上下文？ → A: 是，完整消息历史传递给新的高级执行器。
- Q: 用户是否需要看到当前活跃执行器标识？ → A: 是，始终显示当前活跃执行器的标识。
- Q: 上下文传递失败时如何处理？ → A: 降级传递（截断早期历史，保留最近消息），通知用户并继续切换。

## 用户场景与测试

### 用户故事 1 - 默认低成本对话（优先级：P1）

作为用户，我通过 OpenClaw 现有的低成本路径继续日常多渠道对话。普通交互不受任何影响——消息由默认执行器照常处理。

**优先级原因**: 这是基线体验。混合功能不得导致默认路径出现退化，该路径承载了当前 100% 的流量。

**独立测试**: 发送一条普通消息，不请求任何高级执行器。验证响应来自默认低成本路径，无额外延迟或成本。

**验收场景**:

1. **假设**用户处于正常对话中，**当**用户发送消息且未提及 Claude Code 或 Codex CLI，**则**消息被路由到默认低成本执行器，用户收到与以往相同的响应。
2. **假设**用户在当前会话中从未激活过高级执行器，**当**用户跨多个渠道（如 Slack、Web）进行交互，**则**所有消息继续通过默认路径处理。

---

### 用户故事 2 - 激活高级执行器（优先级：P1）

作为用户，我在对话中显式请求使用 Claude Code 或 Codex CLI。OpenClaw 切换到所请求的高级执行器，该会话中所有后续消息均由该执行器处理。

**优先级原因**: 这是核心新能力——没有它，该功能就没有价值。

**独立测试**: 在对话中显式请求 Claude Code。验证下一条消息及所有后续消息均由 Claude Code 处理，直到用户切换回来。

**验收场景**:

1. **假设**用户处于默认对话中且已有若干轮对话，**当**用户显式请求"使用 Claude Code"，**则** OpenClaw 将当前会话的完整消息历史传递给 Claude Code，激活会话并将后续消息路由到该会话。
2. **假设**用户处于默认对话中，**当**用户显式请求"使用 Codex CLI"，**则** OpenClaw 将完整消息历史传递给 Codex CLI，激活会话并将后续消息路由到该会话。
3. **假设**高级执行器会话已激活，**当**用户发送后续消息，**则**这些消息继续由同一个高级执行器会话处理（保持上下文连续性）。

---

### 用户故事 3 - 停用高级执行器（优先级：P2）

作为用户，我在使用高级执行器后显式请求返回默认路径。OpenClaw 终止高级执行器会话并恢复到低成本路径。

**优先级原因**: 用户需要一个明确的退出路径来控制成本并返回正常操作。

**独立测试**: 激活一个高级执行器，然后显式请求切换回来。验证后续消息通过默认路径处理。

**验收场景**:

1. **假设**高级执行器会话已激活，**当**用户显式请求"切换回来"或"退出 Claude Code"，**则**高级执行器中产生的完整对话内容被带回默认执行器，高级执行器会话被终止，后续消息被路由到默认路径。
2. **假设**高级执行器会话已激活，**当**用户显式请求切换到另一个高级执行器，**则**当前会话的完整消息历史（含高级执行器中的对话）被传递给新执行器，当前会话被终止，新会话启动。

---

### 用户故事 4 - 跨消息的会话持久性（优先级：P2）

作为已激活高级执行器的用户，我的执行器会话在同一 OpenClaw 会话的多条消息之间持续存在。我不需要为每条消息重新激活执行器。

**优先级原因**: 没有持久性，用户体验将严重下降——上下文丢失，且因重复建立会话导致成本增加。

**独立测试**: 激活一个高级执行器，发送 3 条以上消息，验证所有消息均由同一个执行器会话处理且共享上下文。

**验收场景**:

1. **假设** Claude Code 会话已激活，**当**用户在一段时间内发送多条消息，**则** Claude Code 会话在所有消息之间保持其上下文和状态。
2. **假设**高级执行器会话已激活，**当** OpenClaw 会话结束（如超时或用户登出），**则**高级执行器会话也被清理。

---

### 边界情况

- 当用户请求的高级执行器不可用或无法连接时会发生什么？系统应通知用户并保持在默认路径上。
- 如果高级执行器会话在对话中途崩溃会发生什么？系统应通知用户，清理失败的会话，并回退到默认路径。
- 如果用户发送的消息模棱两可，可能是也可能不是在请求执行器切换，会发生什么？系统应仅在明确、无歧义的请求下才进行切换——不应因随意提及而切换。
- 如果同一会话中有多个渠道处于活动状态，且用户从其中一个渠道激活了高级执行器，会发生什么？高级执行器应用于整个会话，而非按渠道生效。
- 如果上下文传递失败（如消息历史过大、目标服务拒绝）会发生什么？系统应降级传递（截断早期历史，保留最近消息），通知用户传递不完整，并继续完成切换。

## 需求

### 功能需求

- **FR-001**: 系统必须将所有消息通过默认低成本执行器路由，除非高级执行器被显式激活。
- **FR-002**: 系统必须支持在对话会话中显式激活 Claude Code 作为高级执行器。
- **FR-003**: 系统必须支持在对话会话中显式激活 Codex CLI 作为高级执行器。
- **FR-004**: 系统必须在所有后续消息中维持已激活的高级执行器会话，直到用户显式停用。
- **FR-005**: 系统必须支持显式停用高级执行器，恢复到默认低成本路径。
- **FR-006**: 系统必须在父级 OpenClaw 会话结束时清理高级执行器会话。
- **FR-007**: 系统必须在高级执行器激活失败时通知用户，并保持在默认路径上。
- **FR-008**: 系统必须仅在用户发出明确、无歧义的请求时才切换执行器——不得因随意提及执行器名称而切换。
- **FR-009**: 系统必须允许从一个高级执行器直接切换到另一个，无需先手动恢复到默认路径。
- **FR-010**: 无论哪个执行器处于活动状态，OpenClaw 都必须始终作为所有交互的编排层。
- **FR-011**: 系统在激活高级执行器时，必须将当前会话的完整消息历史传递给目标执行器。
- **FR-012**: 系统在从高级执行器回退到默认执行器时，必须将高级执行器中产生的完整对话内容带回默认执行器。
- **FR-013**: 系统在高级执行器之间直接切换时，必须将完整消息历史（含当前高级执行器的对话）传递给新目标执行器。
- **FR-014**: 系统必须始终向用户显示当前活跃执行器的标识。
- **FR-015**: 当上下文传递失败时，系统必须降级为截断早期历史、保留最近消息的方式完成传递，通知用户传递不完整，并继续完成执行器切换。

### 关键实体

- **会话（Session）**: 一个 OpenClaw 对话会话，跟踪用户当前状态，包括哪个执行器处于活动状态。
- **执行器（Executor）**: 处理用户消息的后端（默认低成本、Claude Code 或 Codex CLI）。
- **执行器会话（Executor Session）**: 与高级执行器的持久连接/上下文，存在于 OpenClaw 会话之内。

## 成功标准

### 可衡量的成果

- **SC-001**: 用户可以在单次对话轮次内激活高级执行器（无需多步设置）。
- **SC-002**: 用户可以在单次对话轮次内停用高级执行器并返回默认路径。
- **SC-003**: 未激活执行器的会话中，100% 的消息通过默认低成本路径路由（零成本泄漏）。
- **SC-004**: 高级执行器会话在至少 20 条连续消息中保持上下文连续性，不发生会话丢失。
- **SC-005**: 当高级执行器不可用时，用户在 5 秒内收到通知并回退到默认路径。
- **SC-006**: 用户在任何执行器切换（升级、降级、互切）后，不丢失之前的对话内容。
- **SC-007**: 用户在任何时刻都能看到当前活跃执行器的标识。
- **SC-008**: 当上下文传递降级时，用户收到明确的通知说明传递不完整。

## 假设

- OpenClaw 已有可用的多渠道对话系统和会话管理功能。
- Claude Code 和 Codex CLI 作为独立可调用的服务，具有各自的会话管理能力。
- 默认低成本路径保持不变，作为基线执行器。
- 高级执行器的激活以会话为单位，而非以会话内的渠道为单位。
- 不存在自动提升到高级执行器的机制——所有切换均由用户主动发起。
- 高级执行器使用的成本追踪和计费不在本功能范围内（由现有基础设施或单独功能处理）。
